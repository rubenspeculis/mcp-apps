# mcpapps_bridge

The Dart package (consumed via `pubspec.yaml`, not npm) that connects a Flutter component to the host through `@mcpapps/client-core`.

## API

```dart
import 'package:mcpapps_bridge/mcpapps_bridge.dart';

void main() => runMcpApp(const MyApp());
```

- `runMcpApp(Widget child)` — boots the engine, wires the bridge, mounts `child`, then runs the `ui/initialize` handshake (which also sizes the Flutter view from the host's `containerDimensions`, or 480px if the host gives no size).
- `McpApp.of(context)` → `McpAppController`:
  - `.result` — the latest tool result as `Map<String, dynamic>?` (decode with a generated model).
  - `.colorScheme` — `'light'` / `'dark'`.
  - `.callTool(name, args)` → `Future<Map<String, dynamic>?>`.
  - `.requestDisplayMode('fullscreen')` / `.requestDisplayMode('inline')` — asks the host to change display mode.
  - `.initialize()` / `.reportSize(w, h)` — lifecycle, normally driven by `runMcpApp` and `McpAutoSize`.
- `McpAutoSize(child:)` — wraps tall vertical scrollables, measures `maxScrollExtent + viewportDimension`, and reports the real content height to the host.
- `McpAppHttpClient(controller, toolName: '...')` — a `package:http` client that converts HTTP requests into `callTool` calls, useful for inlined `srcdoc` Flutter components where direct browser fetch/XHR is unreliable.

## Under the hood

`McpAppController` is a `ChangeNotifier` provided via an `InheritedNotifier`, so widgets that read `McpApp.of(context)` rebuild on every host update. It talks to the in-iframe `window.mcpappsHost` glue (a thin `client-core` wrapper) over `dart:js_interop`, exchanging JSON strings — the same JSON-bridge pattern, now routed to `postMessage`.

## Tall component sizing

Flutter's canvas does not expose intrinsic DOM height. Without an explicit report, the glue uses `containerDimensions.height`, `containerDimensions.maxHeight`, or a 480px fallback. For tall content:

```dart
McpAutoSize(
  child: SingleChildScrollView(
    child: ResultsView(),
  ),
)
```

`McpAutoSize` reports grow-only by default to avoid feedback loops. Use `growOnly: false` if the component should shrink after content changes.

## Backend calls from inlined Flutter

Inlined Flutter components are commonly rendered through iframe `srcdoc`, which gives the document an opaque origin. Direct browser fetch/XHR can fail or hang even with CORS and `connectDomains` set. Route backend work through server tools instead:

```dart
final client = McpAppHttpClient(
  McpApp.of(context),
  toolName: 'run_projection',
);
final response = await client.post(
  Uri.parse('https://mcp.local/projection'),
  headers: {'content-type': 'application/json'},
  body: jsonEncode({'age': 42}),
);
```

The default adapter sends a JSON object body as the tool arguments and returns the tool's `structuredContent` as a JSON response body. Use `argumentsBuilder` to include request method/path/query data or to map multiple endpoints to one tool.

Pair it with [`@mcpapps/typegen`](/packages/typegen) for typed models instead of raw `Map` access.
