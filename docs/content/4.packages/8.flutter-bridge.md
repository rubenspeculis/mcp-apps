# mcpapps_bridge

The Dart package (consumed via `pubspec.yaml`, not npm) that connects a Flutter component to the host through `@mcpapps/client-core`.

## API

```dart
import 'package:mcpapps_bridge/mcpapps_bridge.dart';

void main() => runMcpApp(const MyApp());
```

- `runMcpApp(Widget child)` — boots the engine, wires the bridge, mounts `child`, then runs the `ui/initialize` handshake (which also sizes the Flutter view from the host's `containerDimensions`).
- `McpApp.of(context)` → `McpAppController`:
  - `.result` — the latest tool result as `Map<String, dynamic>?` (decode with a generated model).
  - `.colorScheme` — `'light'` / `'dark'`.
  - `.callTool(name, args)` → `Future<Map<String, dynamic>?>`.
  - `.initialize()` / `.reportSize(w, h)` — lifecycle, normally driven by `runMcpApp`.

## Under the hood

`McpAppController` is a `ChangeNotifier` provided via an `InheritedNotifier`, so widgets that read `McpApp.of(context)` rebuild on every host update. It talks to the in-iframe `window.mcpappsHost` glue (a thin `client-core` wrapper) over `dart:js_interop`, exchanging JSON strings — the same JSON-bridge pattern, now routed to `postMessage`.

Pair it with [`@mcpapps/typegen`](/packages/typegen) for typed models instead of raw `Map` access.
