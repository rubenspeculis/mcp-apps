# Flutter components

A Flutter MCP App component is a Flutter Web app that reads the host bridge through the `mcpapps_bridge` Dart package.

## Boot and read

```dart [lib/main.dart]
import 'package:flutter/material.dart';
import 'package:mcpapps_bridge/mcpapps_bridge.dart';
import 'generated/models.dart'; // typed model from your zod schema

void main() => runMcpApp(const WeatherApp());

class WeatherApp extends StatelessWidget {
  const WeatherApp({super.key});
  @override
  Widget build(BuildContext context) {
    final raw = McpApp.of(context).result;
    if (raw == null) return const Text('Waiting…');
    final data = GetWeatherOutput.fromJson(raw);
    return Text('${data.tempC.round()}° ${data.condition}');
  }
}
```

| API | Purpose |
| --- | --- |
| `runMcpApp(widget)` | Boots the engine, wires the bridge, runs the `ui/initialize` handshake. |
| `McpApp.of(context)` | The controller: `.result`, `.colorScheme`, `.callTool(name, args)`, `.reportSize(w, h)`, `.requestDisplayMode(mode)`. |
| `McpAutoSize(child:)` | Helper for tall scrollable Flutter content. Reports content height instead of staying clipped at the 480px fallback. |
| `McpAppHttpClient` | `package:http` client that proxies requests through a server tool via `callTool`. |
| `GetWeatherOutput.fromJson` | Typed model generated by [`@mcpapps/typegen`](/packages/typegen). |

## How it connects

`@mcpapps/flutter` runs `flutter build web`, then a small `mcpapps-host.js` glue (the `client-core` bridge) is included in the document. The Dart side calls into that glue via `dart:js_interop`, exchanging JSON strings — so Dart talks to the host through the **same bridge** Vue uses. On boot the bridge runs the handshake and reports size from the host's `containerDimensions`.

## Sizing tall content

Flutter Web renders into a canvas, so the browser cannot infer the component's natural DOM height. The glue starts with the host's `containerDimensions.height`, `maxHeight`, or a 480px fallback. Tall UIs should either call `reportSize` when their content changes or wrap a vertical scrollable with `McpAutoSize`:

```dart [lib/main.dart]
class CalculatorApp extends StatelessWidget {
  const CalculatorApp({super.key});

  @override
  Widget build(BuildContext context) {
    return const MaterialApp(
      home: Scaffold(
        body: McpAutoSize(
          child: SingleChildScrollView(child: CalculatorView()),
        ),
      ),
    );
  }
}
```

`McpAutoSize` listens for `ScrollMetricsNotification` and reports `maxScrollExtent + viewportDimension`. The JS glue grows the Flutter document height before sending `ui/notifications/size-changed`, so the host iframe can expand. It reports grow-only by default to avoid resize loops; pass `growOnly: false` if your component must shrink.

A tall component can also ask for fullscreen when the host supports it:

```dart
await McpApp.of(context).requestDisplayMode('fullscreen');
```

## Inlining for Claude

Hosts serve a single self-contained resource, so for deploy the build is **inlined into one HTML** (`buildFlutterComponent({ inline: true })`), via `@mcpapps/flutter`'s `inlineFlutterBuild`:

- scripts inline; CanvasKit's `.wasm` + fonts are served by an in-document `fetch` shim (synthesized `Response`, no network);
- CanvasKit's dynamic `import()` is redirected by a runtime import-map to a `data:` module;
- only the chromium CanvasKit variant is kept (~10MB).

The result is too large for the Worker script, so `pre-deploy.ts` writes it as a Cloudflare **static asset** and the Worker returns it from `resources/read` via the `ASSETS` binding (`CompiledComponent.htmlAsset`). See [Deploy to Cloudflare](/guides/deploy-cloudflare).

## Networking from inlined Flutter

Claude and similar hosts render the single inlined HTML resource with iframe `srcdoc`. A `srcdoc` iframe has an opaque origin: relative URLs do not point at your Worker, and browser fetch/XHR can be unreliable even when CORS is correct and the origin is listed in `connectDomains`.

For backend work, prefer routing through a server tool with `callTool`. If your Dart code already uses `package:http`, use `McpAppHttpClient` to adapt requests to a tool call:

```dart [lib/api.dart]
import 'dart:convert';

import 'package:flutter/widgets.dart';
import 'package:http/http.dart' as http;
import 'package:mcpapps_bridge/mcpapps_bridge.dart';

Future<Map<String, dynamic>> runProjection(
  BuildContext context,
  Map<String, dynamic> input,
) async {
  final http.Client client = McpAppHttpClient(
    McpApp.of(context),
    toolName: 'run_projection',
  );

  final response = await client.post(
    Uri.parse('https://mcp.local/projection'), // URL is for your API client only.
    headers: {'content-type': 'application/json'},
    body: jsonEncode(input),
  );
  return jsonDecode(response.body) as Map<String, dynamic>;
}
```

By default, a JSON object request body is passed as the tool arguments and the tool's `structuredContent` is returned as a JSON response body. Use `argumentsBuilder` if you need to include the method, path, query, or headers in the tool arguments.

`connectDomains` is still useful for resources that the host allows the component to fetch directly, such as Flutter's font fallback. It is not a substitute for tool-proxied backend calls in `srcdoc` hosts.

## CSP

Flutter web fetches Noto fallback fonts from `fonts.gstatic.com` at runtime, so
`buildFlutterComponent` declares it in the component's `_meta.ui.csp` by default. Add other
origins your app contacts via the `csp` option, but prefer `callTool`/`McpAppHttpClient` for backend requests from inlined Flutter.

## Requirements

- Needs the Flutter SDK (auto-detected via `FLUTTER_BIN`, your `PATH`, or fvm).
- WebAssembly must be permitted by the host (Claude's sandbox allows it). See [renderers](/concepts/renderers) for the trade-offs.

## Run it

```bash [Terminal]
pnpm start    # builds the Flutter web bundle and serves the emulator on :5189
```
