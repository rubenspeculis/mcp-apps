# Deploy to Cloudflare Workers

Because the server is a fetch-native Hono app, deploying to Cloudflare Workers is the natural target.

## The Worker entry

A scaffolded project includes a Workers entry — the same Hono app, no Node APIs:

```ts [src/worker.ts]
import { Hono } from "hono";
import { mountMcp } from "@mcpapps/server/hono";
import { app } from "./app";

const hono = new Hono();
mountMcp(hono, app);
export default hono;
```

## wrangler config

```toml [wrangler.toml]
name = "weather-app"
main = "src/worker.ts"
compatibility_date = "2024-12-01"
compatibility_flags = ["nodejs_compat"]
```

## Deploy

```bash [Terminal]
pnpm deploy            # mcpapps build (compile components) + wrangler deploy
# or directly:
wrangler deploy
```

`mcpapps deploy` compiles your components first, then runs `wrangler deploy`. Point a real MCP host at `https://<your-worker>.workers.dev/mcp`.

## Flutter: static-asset serving

A Flutter project's deploy (`pnpm deploy` → `tsx src/pre-deploy.ts && wrangler deploy`) inlines the build into one ~10MB HTML and writes it as a **static asset**, then the Worker returns it from `resources/read` via the `ASSETS` binding — so the Worker script stays tiny. Its `wrangler.jsonc` therefore declares:

```jsonc [wrangler.jsonc]
{
  "main": "src/worker.ts",
  "compatibility_date": "2025-09-01",
  "compatibility_flags": ["nodejs_compat"],
  "assets": { "directory": "dist", "binding": "ASSETS" }
}
```

`mountMcp` reads `component.htmlAsset` from that `ASSETS` binding automatically. See the [Flutter renderer guide](/guides/flutter-renderer).

::note
The same code also runs on Node (`@hono/node-server`), Bun and Deno — Workers is the default, not a lock-in. To test inside a real host before deploying, use the [Cloudflare tunnel](/guides/tunnel).
::
