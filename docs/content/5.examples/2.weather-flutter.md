# Weather (Flutter)

The `weather-flutter` example is the **same** `get_weather` tool — same server, same emulator, same protocol — rendered by Flutter CanvasKit instead of Vue. It's the clearest demonstration that the renderer is pluggable.

## Run it

```bash [Terminal]
pnpm --filter @mcpapps/example-weather-flutter start   # or: pnpm dev:flutter
```

(Needs Flutter — auto-detected via `FLUTTER_BIN`, your `PATH`, or fvm.) It builds the Flutter web bundle, then serves the emulator on `http://localhost:5189`.

## What it demonstrates

- A typed Dart model (`GetWeatherOutput`) generated from the *same* `zod` output schema by [`@mcpapps/typegen`](/packages/typegen).
- The Dart ↔ host bridge ([`mcpapps_bridge`](/packages/flutter-bridge)) over `dart:js_interop`, consuming the same `client-core` bridge as Vue.
- Single-file delivery for Claude: `pnpm deploy` inlines the whole CanvasKit build into one self-contained HTML and serves it as a Cloudflare static asset (read via the Worker `ASSETS` binding). It renders live at `weather-flutter.otivo.workers.dev`.
- The same `tool-result` → render and `callTool` → re-render round-trip, in Flutter.

Because both examples share the server, emulator and protocol, switching renderers is a project-level choice — not a rewrite.
