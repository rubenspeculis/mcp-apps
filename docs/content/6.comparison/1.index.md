# @mcpapps vs Skybridge

::warning{title="Not affiliated"}
**@mcpapps is an independent project. It is not affiliated with, sponsored by, or endorsed by Skybridge or [Alpic](https://alpic.ai).** "Skybridge" and any related marks belong to their respective owners. This page compares the two frameworks honestly and is written from the @mcpapps point of view.
::

## Inspired by Skybridge

[Skybridge](https://github.com/alpic-ai/skybridge) deserves real credit: it pioneered an excellent developer experience for **MCP Apps**, showed that a typed, hot-reloading workflow against the MCP Apps / mcp-ui standard was possible, and helped popularize interactive UIs returned from MCP tools. @mcpapps was inspired by that work.

@mcpapps takes the same standard in a deliberately **less opinionated** direction. The goal isn't to be "better than" Skybridge — it's to offer **greater freedom** along the axes where Skybridge makes a strong choice for you. If Skybridge's defaults fit your stack, it's a great tool. If you want a different UI framework, server, or runtime, @mcpapps is built for that.

## Greater freedoms

| | Skybridge | @mcpapps |
| --- | --- | --- |
| **UI framework** | React | **Vue or Flutter**, chosen per app (pluggable renderer) |
| **Server** | Express | **Hono** — a fetch-native handler |
| **Runtime** | Node | **Cloudflare Workers, Node, Bun, Deno** from one handler |
| **Renderer model** | React components | A `RendererAdapter` over one shared host bridge — add your own |
| **Type safety** | tRPC-style React inference | One `zod` schema → server validation + Vue inference + **generated Dart models** |
| **MCP Apps / mcp-ui standard** | ✓ | ✓ — same contract, so apps run in Claude, ChatGPT, VS Code |
| **Dev experience** | emulator, hot reload, tunnel | host emulator, live-reload, **Cloudflare tunnel** |
| **License** | open source | open source (MIT) |

::note
Both frameworks implement the same underlying **MCP Apps / mcp-ui** contract. Whichever you choose, your apps target the same hosts — the difference is in the stack you build them with.
::

## When @mcpapps is the better fit

- You want to build MCP App UIs in **Vue** or **Flutter** rather than React.
- You're deploying to **Cloudflare Workers** (or Bun/Deno) and want a fetch-native server.
- You want a **per-app renderer choice**, or to add a renderer of your own against a small bridge interface.
- You value end-to-end types that reach **beyond TypeScript** into Dart.

## When to reach for Skybridge

- Your team is all-in on **React** and a Node/Express backend, and you want a mature, batteries-included experience tuned for that stack.

Either way, you're building on the open MCP Apps standard — and that's a good place to be.
