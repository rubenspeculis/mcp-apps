# Renderers

A renderer turns your component code into a `ui://` resource the host can render. @mcpapps ships two, and you choose **one per app**.

## Vue — self-contained

The Vite plugin compiles a `.vue` SFC + the Vue runtime + `client-core` into **one self-contained HTML document** with everything inlined (JS, CSS, small assets as data URIs). No external network requests — CSP-friendly inside the sandbox. A typical Vue component is tens of KB.

Best for lightweight, fast-loading cards, forms, and dashboards.

## Flutter — inlined to one document

`flutter build web` produces a multi-megabyte CanvasKit bundle (`main.dart.js`, the engine, fonts). Since hosts only serve a single self-contained resource (no sibling assets), `@mcpapps/flutter` **folds the whole build into one HTML document** (`buildFlutterComponent({ inline: true })`): scripts inline, the `.wasm`/fonts served from an in-document `fetch` shim, and CanvasKit's dynamic `import()` redirected by a runtime import-map. The Dart side talks to `client-core` through `dart:js_interop`.

That document is ~10MB, so on Cloudflare it's written as a **static asset** and the Worker returns it from `resources/read` via the `ASSETS` binding (keeping the Worker script tiny). Hosts commonly render the returned HTML with iframe `srcdoc`, so Flutter apps should use `McpAutoSize` for tall canvas content and route backend calls through `callTool`/`McpAppHttpClient` instead of direct browser fetch. See the [Flutter renderer guide](/guides/flutter-renderer). Best for rich, stateful, graphical UIs — charts, maps, canvas — or teams with existing Dart UI.

## The honest trade-off

| | Vue | Flutter |
| --- | --- | --- |
| Bundle size | tens of KB | ~10 MB (CanvasKit, inlined) |
| Cold start | ~tens of ms | hundreds of ms – ~1s |
| Delivery | inlined `srcdoc` | one inlined doc, served as a static asset |
| Sweet spot | lightweight cards | rich/stateful/graphical |

Both run in Claude: Vue is far lighter; Flutter is full-fidelity but heavy (the host re-reads the ~10MB document per render). Pick the renderer that fits the UI.

::note
Both renderers consume the **same** `HostBridge`, so the server, the emulator, the protocol, and your tool definitions are identical regardless of which you choose.
::
