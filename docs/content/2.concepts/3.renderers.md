# Renderers

A renderer turns your component code into a `ui://` resource the host can render. @mcpapps ships two, and you choose **one per app**.

## Vue — self-contained

The Vite plugin compiles a `.vue` SFC + the Vue runtime + `client-core` into **one self-contained HTML document** with everything inlined (JS, CSS, small assets as data URIs). No external network requests — CSP-friendly inside the sandbox. A typical Vue component is tens of KB.

Best for lightweight, fast-loading cards, forms, and dashboards.

## Flutter — asset-bundled

`flutter build web` produces a multi-megabyte CanvasKit bundle (`main.dart.js`, the engine, fonts). That can't be inlined into a single `srcdoc` string, so a Flutter component is served as a **small loader HTML + cached assets**, loaded via an iframe `src`. The Dart side talks to `client-core` through `dart:js_interop`.

Best for rich, stateful, graphical UIs — charts, maps, canvas — or teams with existing Dart UI.

## The honest trade-off

| | Vue | Flutter |
| --- | --- | --- |
| Bundle size | tens of KB | ~1.5–2.5 MB (CanvasKit) |
| Cold start | ~tens of ms | hundreds of ms – ~1s |
| Delivery | inlined `srcdoc` | cached assets via `src` |
| Engine reuse across views | n/a | **not possible** — each MCP-App component is an isolated iframe |
| Sweet spot | lightweight cards | rich/stateful/graphical |

Flutter's CanvasKit is cached immutably and shared across an app's components, so the cost is paid roughly once per session — but the first paint is still heavier than Vue. Pick the renderer that fits the UI.

::note
Both renderers consume the **same** `HostBridge`, so the server, the emulator, the protocol, and your tool definitions are identical regardless of which you choose.
::
