# The host bridge

`@mcpapps/client-core` is the **only** code in the framework that touches `window.postMessage`. Every renderer — Vue, Flutter, and any future one — sits on top of it, so they never diverge in how they talk to the host.

## `HostBridge`

```ts
interface HostBridge {
  onToolResult<T>(cb: (env: ToolResultEnvelope<T>) => void): Unsubscribe;
  onToolInput(cb: (args: Record<string, unknown>) => void): Unsubscribe;
  onTheme(cb: (theme: ThemeState) => void): Unsubscribe;
  onHostContext(cb: (ctx: HostContext) => void): Unsubscribe;
  getLatestToolResult<T>(): ToolResultEnvelope<T> | null;
  getTheme(): ThemeState;
  getHostContext(): HostContext | null;

  callTool<I, O>(name: string, args: I): Promise<ToolResultEnvelope<O>>;
  requestDisplayMode(mode): Promise<void>;   // ui/request-display-mode
  sendMessage(text: string): Promise<void>;  // ui/message
  reportSize(width: number, height: number): void;  // ui/notifications/size-changed

  initialize(): Promise<InitializeResult>;   // runs the handshake; resolves with hostContext
  dispose(): void;
}
```

## Lifecycle handshake

The bridge follows the MCP Apps spec (`2026-01-26`). On mount it sends a `ui/initialize`
**request**, receives the host's `hostContext` (theme, `containerDimensions`, `toolInfo`),
then sends a `ui/notifications/initialized` notification. The host MUST NOT push
`tool-input`/`tool-result` before it sees `initialized`. `mountApp` runs this for you.

## Sizing

Hosts give the iframe flexible dimensions (e.g. Claude: `width:100%`, no fixed height), so
the app **must report its size** or the iframe collapses to 0px. `mountApp` attaches a
`ResizeObserver` to the root and calls `reportSize` (`ui/notifications/size-changed`) on every
change — Vue gets this for free. Flutter starts from the host's
`containerDimensions` or a 480px fallback because the canvas has no intrinsic DOM height;
tall Flutter components should use `McpAutoSize` or call `reportSize` as their content
changes.

## Late-result replay

A Flutter engine can take a moment to boot — long enough that the host's `tool-result` notification arrives **before** the component subscribes. `client-core` caches the latest result and **replays it on subscribe**, so a slow renderer never misses the data. The same mechanism powers the dev server's live-reload: after a hot reload, the last result is replayed and the component re-renders with no manual re-invoke.

## Request correlation

`callTool` generates a JSON-RPC id, sends the request, and resolves the matching response — all hidden from the renderer. A tool call also publishes its result to `onToolResult` subscribers, so a component that reads `useToolResult` updates reactively after a refresh, without threading the return value through your own state.

## Why this matters

The bridge is the reason a Flutter component and a Vue component behave identically against the host: they consume the *same* normalized events. It's also why the [emulator](/guides/emulator) can stand in for a real host — it speaks the exact same protocol, and the bridge can't tell the difference.
