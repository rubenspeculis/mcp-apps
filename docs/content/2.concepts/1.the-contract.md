# The MCP Apps contract

@mcpapps implements the standard **MCP Apps** / [mcp-ui](https://mcpui.dev) contract. Understanding it makes the framework's design obvious.

## The four moving parts

1. **A tool with a UI binding.** A tool advertises a UI resource in its `_meta`:
   ```json
   { "_meta": { "ui": { "resourceUri": "ui://weather-app/get_weather" } } }
   ```

2. **A `ui://` resource.** The host reads that URI via `resources/read` and receives an **HTML document** (`mimeType: text/html;profile=mcp-app`).

3. **A sandboxed iframe.** The host renders the HTML in a `sandbox`ed iframe — no access to the host page, cookies, or storage.

4. **JSON-RPC over `postMessage`.** The iframe and host exchange messages:
   - **host → app:** `ui/notifications/tool-result` (the tool's output), `ui/notifications/theme`.
   - **app → host:** `tools/call` (invoke another tool), display-mode and follow-up-prompt requests.

## How @mcpapps maps onto it

| Contract piece | @mcpapps |
| --- | --- |
| Tool + `_meta.ui` | `defineTool({ ui })` on `@mcpapps/server` |
| `ui://` HTML resource | compiled by `@mcpapps/vite-plugin-vue` (Vue) or `@mcpapps/flutter` |
| iframe ↔ host messaging | `@mcpapps/client-core` (the only `postMessage` peer) |
| `structuredContent` | your tool's typed return, validated against its `zod` output schema |

Constants for every wire-level value (the mime string, method names, the `_meta` shape) live in `@mcpapps/protocol` — a single source of truth, so a spec change is a one-line edit.

::note
Because @mcpapps speaks the real standard, your apps run in hosts that implement MCP Apps (Claude, ChatGPT, VS Code) without modification.
::
